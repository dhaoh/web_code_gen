## 一、解决了什么问题

现在大语言模型（比如 DeepSeek、GPT-4）可以直接从自然语言描述生成代码，这个能力看起来很强大，但真正用到工程里，问题就暴露出来了。

我在工作中对此有比较深的感受。公司前段时间做一个量化交易平台，核心功能是让用户用自然语言描述交易逻辑——比如"当 5 日均线上穿 20 日均线时买入，当跌破 10 日均线时卖出"——然后由 LLM 自动生成对应的量化策略代码，再到历史 A 股数据上回测，最终输出该策略的绩效曲线图。这个功能听起来很美好，但实际开发中我们发现：**哪怕用户前后两次输入完全相同的自然语言需求，LLM 生成的策略代码也不一样。** 有时候换了一种变量命名，有时候调了一个 API 的顺序，更严重的时候直接生成了一段语法正确但逻辑完全错误的代码，回测结果南辕北辙，整个功能就不可用了。

这个亲身经历让我意识到，纯靠 LLM"裸生成"代码，在工程上至少有四个致命问题：

**第一，输出不稳定。** 同样输入，每次输出都不同。我们实验测出来，同样一个学生选课系统的需求让 LLM 生成三次，三次的文件结构一致性只有 48%。对于量化策略来说，这种不确定性意味着同一个用户的同一条策略，回测结果可能天差地别——这在金融场景里是不可接受的。

**第二，代码编译不过。** LLM 虽然能把单段代码写对，但跨文件之间的依赖关系经常处理不好——比如引用不存在的模块、在错误的目录结构里用相对导入。我们实验里纯 LLM 方法生成的代码 100% 导入失败，可编译率只有 33%。

**第三，业务规则容易遗漏。** 比如"选课不能超过容量上限"、"不能重复选同一门课"这些约束，纯 LLM 经常会漏掉或者实现错误。换成量化场景，就是"止损条件没生效"、"仓位管理逻辑丢失"这类问题。

**第四，出了问题没法自动修复。** 纯 LLM 生成完就结束了，代码能不能跑、逻辑对不对，没有任何保障机制。

**我们要解决的核心问题就是：** 怎么把 LLM 的代码生成能力"管"起来——让它生成出结构正确、能编译运行、而且业务逻辑完整无误的代码？

## 二、怎么用这门课的知识解决的

这门课讲的是软件需求分析与设计，核心思想是：**先用模型描述系统的需求和结构，再从模型出发去生成代码。** 我们就是把这个思想用到了极致。

具体来说，把我们课上学的方法跟 LLM 结合起来，做了一个完整的流水线：

**第一步，用 YAML 定义领域模型。** 这个模型包含了所有实体（学生、课程、选课记录）、每个实体的属性（类型、是否必填、是否唯一）、实体之间的关系（一对多、多对多）、以及业务规则（容量检查、重复选课检查）。这就是需求分析阶段的工作。

**第二步，模型转换为代码骨架。** 我们用 Pydantic 把 YAML 解析成强类型的中间表示（IR），然后用 Jinja2 模板引擎从 IR 生成确定性的代码骨架——SQL 建表语句、FastAPI 的 CRUD 路由、React 前端组件等等。架构完全由模型决定，每次生成的结果都一样。这对应了设计阶段。

**第三步，LLM 只负责填业务逻辑。** 骨架里预置了 `# LLM_FILL:` 标记，LLM 看到标记和模型上下文后，只在这些位置插入对应的验证代码。LLM 只管行为逻辑，不管架构。

**第四步，双向反馈机制自动修正。** 代码生成后，自动跑语法检查和导入检查。如果发现错误，只把出错的文件和模型上下文一起发回 LLM 修复，修完再检查，直到通过为止。这是我们的创新点——相当于把测试驱动的思想做成了自动化闭环。

**对比基线**就是纯 LLM 方法——直接把自然语言需求发给 DeepSeek，让它一次性生成全部代码，不加任何结构约束和反馈修正。

## 三、效果比不用这门课的方法好在哪里

用三个数据说话：

| 指标 | 模型驱动方法 | 纯 LLM 方法 | 提升 |
|------|-------------|-------------|------|
| 可编译率 | **100%** | 33.3% | 三倍差距 |
| 一致性 | **98.4%** | 48.1% | 几乎翻倍 |
| 结构正确率 | **100%** | 91.7% | |

**可编译率达到 100%**——语法、导入、前端构建三项全部通过。而纯 LLM 方法虽然语法能过，但导入检查全军覆没。这个差距是最关键的——你生成的代码能不能用，这个是硬指标。

**一致性达到 98.4%**——意味着不管跑多少次，生成的代码结构几乎一模一样。这对团队协作、代码审查、CI/CD 流水线来说非常重要。

**反馈机制 1 轮收敛**——所有模型的错误都在 1 轮反馈内修正完毕。因为 LLM 有模型上下文这个"护栏"，知道修 bug 的时候不能出界，所以修得又准又快。

**本质原因**：模型驱动方法把"架构"和"行为"分开了。架构由确定性的模板生成，永远不错；行为由 LLM 在架构内填充，错了还能自动修。纯 LLM 方法是把两件事一起扔给模型，它分不清哪是结构哪是逻辑，所以出错率高且不稳定。

## 附录：老师可能问的问题 & 简短回答

### Q1：YAML 模型是谁来写的？是不是也要花不少时间？
确实需要人来写。但 YAML 模型的抽象层次比代码高得多——3 个实体的模型只有 30 行 YAML，却能生成 5 个后端文件加 8 个前端文件、总共几百行代码。而且模型写一次可以复用，修改需求时改模型比改代码快得多。

### Q2：如果需求变了，是不是要重新建模、重新生成？
是的，这恰恰是模型驱动方法的优点。需求变了，先改 YAML 模型（比如加一个字段、改一个约束），然后重新跑生成流水线。因为代码骨架是确定性的，只需要 LLM 重新填充改动部分的业务逻辑，其他部分不受影响。

### Q3：你用的是什么 LLM？换成 GPT-4 或 Claude 会不一样吗？
用的是 DeepSeek V3。我们没有对比多个 LLM（这是未来工作方向之一），但理论上，模型驱动方法的核心优势——确定性骨架 + 反馈修正——不依赖特定 LLM，换其他模型只会影响业务逻辑填充的质量，不会影响结构和编译的正确性。

### Q4：只测了后端？还是前端也测了？
前后端都测了。后端测了 Python 语法检查和模块导入，前端测了 TypeScript 编译和 Vite 打包。模型驱动方法三项全过。

### Q5：反馈循环最多只跑 3 轮，万一还是修不好怎么办？
实验中三个模型全部在 1 轮内修好。如果超过 3 轮还没通过，说明错误可能超出了 LLM 的能力范围（比如缺依赖包）。这种情况下标记为失败，在实际使用中就需要人工介入。

### Q6：你的创新点"双向反馈机制"跟现有工作比有什么不同？
关键不同在于三点：一是只修有错误的文件（不是整个工程扔给 LLM），二是带了模型上下文（LLM 知道修的时候有哪些约束），三是限制修改范围（只允许在 LLM_FILL 标记附近修改）。传统 LLM 自调试没有这些约束，修 bug 时容易引入新的结构性问题。

### Q7：Small 模型数据很好，Medium 和 Large 呢？
Medium 和 Large 模型生成器验证都能通过（语法和导入检查全过），但因为 API 稳定性和费用问题，完整的对比实验数据目前只完整收集了 Small 模型。模型驱动方法在不同规模下的表现理论上是一致的——因为骨架是由模板确定生成的，LLM 只填一次 routes.py，规模增大不影响结构正确性和编译性。

### Q8：生成的代码质量和人工写相比怎么样？
我们主要衡量了可编译性和结构正确性这两个硬指标。代码风格、可维护性这些软指标没有量化。但从实验结果来看，模型驱动方法生成的代码通过了语法检查、导入检查、TypeScript 编译和 Vite 构建——至少是能跑起来的代码，这点纯 LLM 方法做不到。

## 附2：关于量化交易经历的预期提问

### Q9：你说的量化策略代码生成，具体 LLM 是怎么生成代码的？是直接输出 Python 吗？
用户输入自然语言描述（比如"当 5 日均线上穿 20 日均线时买入"），系统把这个描述包装成 prompt 发给 LLM，LLM 返回一段完整的 Python 策略代码，代码里会调用我们平台上封装好的数据接口（比如 `get_price()`、`get_ma()`）和交易接口（比如 `buy()`、`sell()`）。然后系统拿到这段代码，注入到回测引擎里，用历史数据跑一遍，最后生成收益曲线、最大回撤等绩效指标。

### Q10：你们当时遇到"不稳定"这个问题后，尝试过什么解决方案？
最开始试过在 prompt 里加更详细的约束——比如指定函数命名规范、要求固定的代码模板结构。效果有一点改善，但不能根治。也考虑过对 LLM 输出做语法检查和沙箱执行，但这些都是"事后补救"，没法从源头保证结构正确性。后来因为项目排期问题没有继续深入。这次做这个课程项目，相当于是把当时想解决但没来得及解决的问题，用我们课上学的模型驱动方法做了一个系统性的方案。

### Q11：你做的这个课程项目的方法，能直接用到你们公司的量化交易场景吗？
思路上完全通用，但细节需要适配。核心逻辑是一样的：先用一个形式化的"领域模型"把量化策略的结构给固定下来——比如策略一定有"选股条件"、"买入信号"、"卖出信号"、"止损规则"、"仓位管理"这几个模块——然后用模板生成策略代码的骨架，LLM 只负责在每个模块内填充具体的交易逻辑。这样不管用户怎么描述，生成的代码结构永远是一致的，就不会出现"同一个需求生成两次结果不同"的问题了。

### Q12：生成量化策略代码的时候，回测结果不对怎么排查？
这个问题本质上也是我们这个反馈机制的思路——把回测结果（比如收益率异常高或低、交易次数为零、或直接报错）作为反馈信号，自动发回 LLM 让它检查并修正策略逻辑。我们的双向反馈循环目前只用了语法和导入检查作为反馈信号，但框架设计上可以把任何可量化的反馈（包括回测指标）加进去。这是未来工作中"Runtime test integration"那一项的具体含义。

### Q13：你们的量化策略项目最后怎么样了？这个课程项目对你的工作有什么启发？
量化策略项目因为代码生成稳定性问题，最后只上线了一个功能受限的版本——只能支持预设好的几种策略模板，用户自定义策略的能力没有达到预期。这个课程项目让我意识到：问题不在于 LLM 不够强，而在于我们没有给 LLM 一个足够好的"结构约束"。把模型驱动工程的方法引入 LLM 代码生成流程，让 LLM 只做它擅长的部分（逻辑填充），而架构由规则来保证——这个思路对我以后做 AI 辅助开发或者代码生成类的产品，有很大的指导意义。
